## Learning Rule Improvement

今天发现：

第一次学习一个新的知识点时，直接预测运行结果，会导致学习效率下降。

因此新增永久教学规则：

所有新知识必须遵循：

原理

↓

字面

↓

真实代码

↓

真实运行结果

↓

总结规律

↓

预测

↓

练习

目的：

培养推理能力，而不是随机猜测代码。

## Tutorial 09 课程设计与 Part 1 学习

### 今天实际完成了什么

Tutorial 08 已完成。今天开始共同设计 Tutorial 09，并将其命名为 `09_structured_agent`，同时创建了 Part 1 的教学材料和代码草稿。

Part 1 的主题是：**结构化 Agent action 与工具调用解析**。

Tutorial 08 使用普通文本协议：

```text
[TOOL:read_file:{"path":"tools.py"}]
```

Part 1 计划改用 JSON action 协议：

```json
{
  "kind": "tool_calls",
  "tool_calls": [
    {"name": "read_file", "arguments": {"path": "tools.py"}},
    {"name": "print", "arguments": {"text": "读取完成"}}
  ]
}
```

新增代码：

- `ToolCall`：表示一个待执行的工具调用
- `AgentAction`：表示 Agent 当前响应是工具调用还是最终回答
- `parse_action()`：把 JSON 文本解析成结构化对象
- `describe_action()`：观察结构化结果，不执行工具

### 代码草稿的运行结果（不计入学习完成）

```text
ACTION KIND: tool_calls
TOOL CALL COUNT: 2
CALL 1: read_file({'path': 'tools.py'})
CALL 2: print({'text': '读取完成'})
```

### 设计目标（不是学生已掌握内容）

Part 1 计划建立第一条新边界：

> 解析 Agent 想做什么，与执行工具，不是同一个职责。

代码草稿演示了一个响应可以被解析为多个独立的 `ToolCall`。今天学生已经通过真实运行、预测和练习验证了该规律。

## Tutorial 09 Part 1 正式学习记录

### 今日学习内容

今天正式学习了 Tutorial 09 Part 1：结构化 Agent action 与多工具调用解析。

核心新概念：

- `AgentAction`：表示 LLM 这一轮想做什么，是最终回答还是工具调用
- `ToolCall`：表示一个具体工具调用，包含工具名和参数
- `parse_action()`：把 LLM 的 JSON 文本解析为 Agent 内部稳定的动作对象

### 今日数据流

```text
LLM JSON 文本
↓
json.loads()
↓
Python dict/list
↓
ToolCall 对象列表
↓
AgentAction
↓
后续执行器再读取并执行
```

### 真实运行结果

原始示例运行后观察到：

```text
ACTION KIND: tool_calls
TOOL CALL COUNT: 2
CALL 1: read_file({'path': 'tools.py'})
CALL 2: print({'text': '读取完成'})
```

练习中新增第三个工具调用后，观察到：

```text
ACTION KIND: tool_calls
TOOL CALL COUNT: 3
CALL 1: read_file({'path': 'tools.py'})
CALL 2: print({'text': '读取完成'})
CALL 3: list_files({'path': '.'})
```

### 今日练习

学生在 `demo_part1.py` 中为 `RAW_RESPONSE` 增加第三个工具调用：

```json
{"name": "list_files", "arguments": {"path": "."}}
```

第一次运行时遇到 JSON 缺少逗号导致的错误：

```text
json.decoder.JSONDecodeError: Expecting ',' delimiter
```

随后修复第二个工具调用和第三个工具调用之间缺少的逗号，并成功运行。

### 当前理解情况

学生能用自己的话解释：

- `AgentAction` 用来把 LLM 文本转换成 Agent 看得懂的动作结构
- `ToolCall` 是一个具体工具调用
- `AgentAction` 是这一轮 Agent 动作的整体表示，可以包含多个 `ToolCall`
- `parse_action()` 负责把 LLM JSON 文本解析为 dict，再转换成 `ToolCall`，最后包装成 `AgentAction`
